# STICS Semantic Adapter with Helio

This repository contains the Docker recipe required to run the **STICS Semantic Adapter**, a service developed in the context of the European project **STICS**.

The adapter uses the **Helio ecosystem** to transform heterogeneous source data into a homogeneous and semantically interoperable representation. Given a set of translation mappings, Helio generates RDF data expressed according to the ontology developed within STICS. In this way, the adapter acts as an interoperability layer between the original data sources and the STICS Data Space connector.

The deployment exposes two components:

- **Helio REST service**, which executes the mappings and publishes the generated RDF data.
- **Helio graphical interface**, which can be used to configure and manage mappings through a web browser.

> Important: the graphical interface is available on port **4201**, while the semantic data generated by the adapter is published through the REST service on port **4567**.

---

## Repository structure

A typical deployment has the following structure:

```text
.
├── docker-compose.yml
├── local/
└── db/
```

Where:

- `docker-compose.yml` contains the Docker recipe for running the adapter.
- `local/` is mounted inside the Helio container and can be used to provide local files required by the mappings.
- `db/` is mounted inside the Helio container and stores the Helio database, including mappings and configuration.

The two folders `local/` and `db/` are mounted as Docker volumes so that their content is persisted outside the container. This means that mappings and configuration are not lost when the service is restarted.

---

## Docker deployment

The service is deployed using Docker Compose.

The provided `docker-compose.yml` defines two services:

```yaml
version: '3'
services:
  helio-rest:
    image: acimmino/helio-rest:latest
    volumes:
      - ./local/:/usr/src/app/local/
      - ./db/:/usr/src/app/db/
    ports:
      - "4567:4567"

  frontend:
    image: emiliocrespoperan/helio-frontend:latest
    depends_on:
      - helio-rest
    ports:
      - "4201:80"
```

### 1. Create the required folders

Before starting the service, create the folders used by the Docker volumes:

```bash
mkdir -p local db
```

### 2. Start the adapter

Run the following command from the folder containing the `docker-compose.yml` file:

```bash
docker compose up -d
```

If your environment still uses the legacy Docker Compose command, use:

```bash
docker-compose up -d
```

### 3. Check that the containers are running

```bash
docker compose ps
```

You should see two running services:

- `helio-rest`
- `frontend`

### 4. Access the services

Once the containers are running, the services are available at:

| Component | URL | Description |
|---|---|---|
| Helio graphical interface | `http://localhost:4201` | Web interface to configure and manage mappings |
| Helio REST service / RDF data publication | `http://localhost:4567` | REST API used to publish and consume semantic interoperable RDF data |

To list the data endpoints currently provided by the Helio REST service, you can use:

```bash
curl http://localhost:4567/api/
```

## How the adapter works

The STICS Semantic Adapter follows this general workflow:

1. **Original data sources** are provided to the adapter.
2. **Translation mappings** describe how the original data must be interpreted and transformed.
3. **Helio executes the mappings** and generates RDF.
4. The resulting RDF uses the **STICS ontology**, providing a homogeneous semantic representation of the data.
5. The RDF data is published through the Helio REST service, making it available for other STICS components.

This allows heterogeneous data to be consumed in a common semantic format, improving interoperability between the different systems and actors involved in the project.

## Adding a new mapping

A mapping defines how a specific input data source is transformed into RDF using the STICS ontology. New mappings can be added through the Helio graphical interface.

### 1. Open the graphical interface

Open the Helio frontend in your browser:

```text
http://localhost:4201
```

The interface is exposed on port **4201**.

### 2. Create or register a new mapping

From the graphical interface, create a new translation task or mapping. The mapping must describe:

- The input data source to be transformed.
- The structure of the input data.
- The RDF resources to be generated.
- The STICS ontology classes and properties used in the output RDF.
- Any required local files, if the mapping depends on files mounted in the `local/` folder.

If the mapping uses local files, place them in the local folder of the host machine:

```text
./local/
```

This folder is mounted inside the Helio REST container at:

```text
/usr/src/app/local/
```

Therefore, mappings executed inside the container should refer to local files using the container path.

### 3. Save the mapping

When the mapping is saved, Helio stores the configuration in the database folder mounted as:

```text
./db/ -> /usr/src/app/db/
```

This ensures that mappings are preserved when containers are stopped or restarted.

### 4. Publish and test the generated RDF

After registering the mapping, the generated RDF data can be accessed through the Helio REST service on port **4567**.

You can inspect the list of available endpoints with:

```bash
curl http://localhost:4567/api/
```

Depending on the identifier assigned to the mapping, the generated data will be available through the corresponding Helio endpoint.

For example:

```bash
curl http://localhost:4567/api/<mapping-id>
```

Replace `<mapping-id>` with the identifier of the mapping created in the interface.

## Updating an existing mapping

To update an existing mapping:

1. Open the Helio graphical interface at `http://localhost:4201`.
2. Locate the existing mapping or translation task.
3. Modify the mapping definition.
4. Save the changes.
5. Test the generated RDF again through the REST service on port `4567`.

The updated configuration will remain persisted in the `db/` folder.

---

## Stopping the service

To stop the adapter without deleting persisted data:

```bash
docker compose down
```

This stops the containers but keeps the `local/` and `db/` folders in the host machine.

To start the service again:

```bash
docker compose up -d
```

## Troubleshooting

### The graphical interface does not load

Check that the frontend container is running:

```bash
docker compose ps
```

Then verify that port `4201` is not being used by another service.

### The RDF data endpoint does not respond

Check that the Helio REST container is running and that port `4567` is exposed:

```bash
docker compose ps
curl http://localhost:4567/api/
```

### Mappings disappear after restarting the service

Make sure that the `db/` folder exists and is mounted correctly:

```bash
ls -la db
```

The Docker Compose file must include this volume:

```yaml
- ./db/:/usr/src/app/db/
```

### Local files are not found by a mapping

Make sure that the files are placed under:

```text
./local/
```

Inside the container, this folder is available at:

```text
/usr/src/app/local/
```

Use the container path when referencing those files from a mapping.
